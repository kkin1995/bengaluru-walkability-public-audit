## Issues Found

### 1. favicon.ico missing (404)

```
nginx: "GET /favicon.ico HTTP/1.1" 404 3289
```

No favicon is served. Browsers always request it; the 404 is harmless but adds noise to logs and shows a broken-icon tab in browsers.

**Fix:** Add a `favicon.ico` (or `icon.png`) to `frontend/app/` — Next.js App Router picks it up automatically.

---

### 2. Next.js metadata warnings

```
⚠ Unsupported metadata themeColor is configured in metadata export in /favicon.ico.
  Please move it to viewport export instead.
⚠ Unsupported metadata viewport is configured in metadata export in /favicon.ico.
  Please move it to viewport export instead.
```

`themeColor` and `viewport` were moved out of `metadata` and into a separate `generateViewport()` export in Next.js 14. They currently live in the wrong export, so Next.js ignores them.

**Fix:** In the relevant layout/page file, split the export:

```ts
// Before (broken)
export const metadata: Metadata = {
  themeColor: '#...',
  viewport: 'width=device-width, initial-scale=1',
  ...
}

// After (correct)
export const metadata: Metadata = { ... }

export const viewport: Viewport = {
  themeColor: '#...',
  width: 'device-width',
  initialScale: 1,
}
```

### 3. nginx buffered photo upload to disk

```
nginx: [warn] a client request body is buffered to a temporary file
       /var/cache/nginx/client_temp/0000000002
       request: "POST /api/reports HTTP/1.1"
```

The photo upload was large enough that nginx couldn't buffer it in memory and spilled it to a temp file. This is **expected behaviour** for image uploads but means:
- Upload latency has a disk I/O component inside nginx
- The temp dir needs adequate disk space in production

**Options:**
- Increase `client_body_buffer_size` in nginx.conf (trades RAM for fewer disk spills)
- Or accept the current behaviour — disk buffering is safe, just slightly slower
